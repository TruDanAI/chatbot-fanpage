# Phase 4 v1 Internal Notes Design

This document defines the first Phase 4 write workflow on paper only. It is
not approval to implement POST routes, UI write forms, production migrations,
production environment changes, production database writes, production `/data`
access, or production admin user creation.

## Goal

Add a small operator-facing write workflow: an admin can add an internal note
to an order, conversation, or customer/sender for later staff context.

The note workflow is intentionally narrow:

- It does not change order status.
- It does not change customer profile fields.
- It does not change conversation or bot runtime state.
- It does not send Messenger messages.
- It does not trigger staff notifications or external integrations.
- It does not expose raw customer rows, raw order rows, or raw message rows.

## Scope

Supported note targets for v1:

- `order`: `target_id` is the string form of `orders.id`.
- `conversation`: `target_id` is the `sender_id`.
- `customer`: `target_id` is the `sender_id`.

The UI/API implementation must validate that `target_type` is one of these
values and that `target_id` is non-empty. If implementation later chooses to
verify target existence before note creation, that check must be parameterized
and tenant/page-scoped.

Note body is operator-authored text. It may contain operational context, but
the implementation must not copy raw messages, raw customer/order data, phone
numbers, addresses, tokens, or secrets into audit metadata or responses.

## RBAC

Phase 4 v1 introduces a narrow write permission:

- `admin.internal_note.write`

Role assignment for v1:

- `maintainer`: can add internal notes.
- `owner`: can add internal notes.
- `support`: remains read-only for internal notes in v1.
- `viewer`: remains read-only.

Support write access can be reviewed later after the first workflow is tested
and production behavior is understood. This permission should stay separate
from broad order mutation permissions such as `admin.order.write`.

## Actor Requirements

Every created note must have a safe resolved actor:

- `created_by` must be populated at implementation time.
- The implementation must not create production notes with empty `created_by`.
- Browser sessions should eventually resolve to a PostgreSQL `admin_users.id`
  actor when real admin identity is implemented.
- Bearer automation should resolve to `automation:admin_export_token`.
- The current browser-session bridge is not proof of per-human identity; until
  identity is implemented, the actor must follow the documented Phase 3.5
  actor semantics.

Avoid raw emails, tokens, cookie values, or display-only names as the durable
actor id.

## Schema Proposal

This SQL is an additive/idempotent proposal only. Do not apply it to
production until a fresh PostgreSQL backup exists, non-production apply is
verified, and the owner separately approves the production schema write.

```sql
CREATE TABLE IF NOT EXISTS internal_notes (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  tenant_id TEXT NOT NULL,
  page_id TEXT NOT NULL,
  target_type TEXT NOT NULL,
  target_id TEXT NOT NULL,
  body TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'visible',
  created_by TEXT NOT NULL DEFAULT '',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  hidden_by TEXT NOT NULL DEFAULT '',
  hidden_at TIMESTAMPTZ,
  hide_reason TEXT NOT NULL DEFAULT '',
  CHECK (tenant_id <> ''),
  CHECK (page_id <> ''),
  CHECK (target_type IN ('order', 'conversation', 'customer')),
  CHECK (target_id <> ''),
  CHECK (body <> ''),
  CHECK (char_length(body) <= 2000),
  CHECK (status IN ('visible', 'hidden'))
);

CREATE INDEX IF NOT EXISTS internal_notes_target_time_idx
  ON internal_notes (tenant_id, page_id, target_type, target_id, created_at DESC, id DESC);

CREATE INDEX IF NOT EXISTS internal_notes_status_time_idx
  ON internal_notes (tenant_id, page_id, status, created_at DESC, id DESC);
```

Implementation should enforce non-empty `created_by` before insert even though
the draft schema uses a default for additive compatibility.

## Audit Semantics

Every internal-note create attempt must be audited.

For successful creates:

- `action = admin.internal_note.create`
- `resource_type = internal_note`
- `resource_id = note_id`
- `outcome = success`

For denied attempts:

- `action = admin.internal_note.create`
- `resource_type = internal_note`
- `resource_id = ''` if no note exists yet
- `outcome = denied`
- metadata may include a safe denial reason and required permission

For implementation/runtime errors:

- `action = admin.internal_note.create`
- `resource_type = internal_note`
- `resource_id = note_id` only if a note id was allocated before the error;
  otherwise leave it empty
- `outcome = error`

Audit metadata may include:

- `target_type`
- `target_id`
- `body_length`
- `auth_method`

Audit metadata must not include:

- note body
- raw message data
- raw customer data
- raw order data
- phone numbers
- addresses
- tokens
- secrets
- database URLs
- cookie values

Write audit must fail closed. During implementation, inserting the internal
note and inserting the audit record must happen in the same database
transaction. If the audit insert fails, note creation must roll back and the
operator must receive an error instead of a partially committed note.

## UI Confirmation Requirement

The v1 UI must require explicit confirmation before creating a note:

- Show the target type and safe target id.
- Show only masked or already-safe context.
- Require a non-empty note body.
- Limit body length to 2000 characters.
- Use an explicit submit label such as `Add internal note`.
- Do not provide bulk note creation.
- Do not provide edit/delete controls in v1.

If a future API endpoint is added before a full UI form, it must still require
the same validation, authorization, audit, and transaction behavior.

## Rollback Stance

Rollback is non-destructive by default:

- Hide notes with `status = 'hidden'`, `hidden_by`, `hidden_at`, and
  `hide_reason`.
- Do not delete notes in v1.
- Do not remove audit rows.
- If code deploy has issues, redeploy the previous known-good commit.
- If schema exists but the feature is disabled or not deployed, leave the
  additive table in place.
- Dropping `internal_notes` is destructive and requires a separate fresh
  backup, restore plan, and explicit approval.

## Tests Required Before Implementation

Before any route/UI implementation is merged, add or update tests for:

- RBAC: `maintainer` and `owner` can create notes; `viewer` and `support` are
  denied.
- Validation: invalid target type, empty target id, empty body, and overlong
  body are rejected.
- Actor handling: note creation rejects an empty or unresolved `created_by`.
- Audit metadata: metadata includes safe target/body length/auth fields and
  excludes note body, raw rows, phone, address, tokens, and secrets.
- Transaction behavior: if audit insert fails, the note insert rolls back.
- Response/UI safety: successful and failed responses do not render raw
  customer/message/order rows.
- Count-only verification helpers for future production smoke.

Local verification before any implementation push/deploy:

- `node --check` for touched JavaScript files.
- `npm test`.
- `npm audit --omit=dev`.
- `git diff --check`.

## Production Rollout Plan

Production rollout must be split into separate approvals:

1. Re-check git, latest Railway deployment, and `/healthz`.
2. Create a fresh production PostgreSQL backup outside the repo using
   read-only SELECT/export.
3. Verify backup SHA256 and count-only summaries.
4. Apply the schema to local/staging first and verify with count-only queries.
5. Request separate owner approval for production schema apply.
6. Apply production schema only after that approval.
7. Verify production schema with count-only checks only.
8. Request separate owner approval for code deploy.
9. Deploy code only after that approval.
10. Run safe smoke first: `/healthz` and GET `/admin/login`.
11. Request separate owner approval before authenticated production smoke,
    because admin auth smoke writes audit rows and note smoke writes the first
    business note.
12. Run one approved note-create smoke against a chosen safe target.
13. Verify only counts and outcome deltas:
    `internal_notes`, visible/hidden counts, and `admin_audit_log` outcomes.

Do not print note bodies, raw customer data, raw order rows, raw message rows,
tokens, secrets, or database URLs during rollout.

## Explicit Non-Goals

- Do not add POST routes as part of this design.
- Do not add UI write forms as part of this design.
- Do not add migrations to production as part of this design.
- Do not change production environment variables.
- Do not write production PostgreSQL.
- Do not touch production `/data`.
- Do not create production admin users.
- Do not build an enterprise workflow engine.
- Do not add bulk updates.
- Do not add delete customer/conversation/order features.
